Consider the C++ header files in spec/Windows/inc. These files make up the standard ODBC 4.0 C interface. The plan is to convert those headers in to DI files for D programming language that can be used to call the ODBC C library directly. As ODBC is cross platform we will need consider what is available on each platform supported by D. Next, we will want to consider the A and W method variants for handling 1-byte and a 2-byte wide characters. In D, C macros can be implemented as template functions. Once the conversion is complete we want to create a D-native wrapper API around the C-native ODBC API. This is due to two factors, first D has native types for strings/wide-strings that need to be converted to match the C-native ODBC API and it would be useful to be able to call the API using D-native types. Second, we want to use a result-type model for return values, so that we can report errors and return values without using Exceptions. D has no native Result type so one will need to be created for this project.

The predefined version identifiers should be sufficient to determine which version of ODBC is in use as each platform will provide it's own version of the driver. For example, a version(OSX) would be sufficient to know that iODBC is in use.
The result type must also include a text buffer to include the specific error message to aid in debugging of user apps as some errors contain information that is specific to the user's database.
